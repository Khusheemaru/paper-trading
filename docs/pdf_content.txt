Face Recognition System 
Cache Issues in Existing System 
•Cache created on every 
request/run. 
•Cache supported only a single 
folder. 
•No automatic update when: 
–Image added 
–Image deleted 
–Image modified 
•Cache not synchronized with 
database. 
•Slow retrieval for large datasets. 
5 Impact 
•Response time in minutes. 
•Unnecessary cache recreation. 
•Incorrect recognition due to 
outdated cache. 
•Poor scalability. 
--- Page 6 ---
Face Recognition System 
Improvements Implemented 
•Designed hierarchical folder-level 
cache system. 
•Auto-detects nested folder 
structure from database. 
•Generates cache for all images 
across multiple levels. 
•Cache created: 
–Once at application startup. 
–Not on every request. 
6 Smart Update Logic 
•Auto-update cache on: 
•Image addition 
•Image update 
•Image deletion 
•No manual GET/PUT request 
required. 
•If full database changes → 
rebuild cache only at startup. 
Results 
•Cache creation time reduced from minutes to few seconds. 
•Retrieval became near real-time. 
•Eliminated redundant cache creation. 
•Ensured database-cache consistency. 
•Improved system scalability and stability.

Optimization Strategies: PPR & ISR 
 -Partial Pre-rendering (PPR): Implemented a hybrid   
rendering model where the static page shell loads 
instantly, while dynamic parts (like view counts) are 
streamed in. 
  
 -Incremental Static Regeneration (ISR): Configured 
background revalidation (e.g., every 60 seconds) to keep 
startup details fresh without slowing down the server. 
 -Data-Cache Consistency: Optimized the retrieval process 
to ensure that the user always sees the most recent 
pitches.
